Datadog SLOs as Terraform

SLOs defined as code — version-controlled, peer-reviewed, deployed through CI/CD. Not click-ops.

Why this matters: The availability SLO tracks 99.9% of requests returning non-5xx over 30 days. That gives us a 43.2 minute error budget per month. The latency SLO ensures p99 stays under 500ms — because every 100ms of added latency costs roughly 1% of marketplace conversion.

Multi-Window Burn-Rate Alerts

Instead of alerting on raw error rates, we alert when the error budget is being consumed too fast.

Two alert tiers: P1 pages when the 1-hour burn rate hits 14.4x (budget gone in ~2 hours). P3 creates a ticket at 6x (budget gone in ~5 days). This avoids noisy alerts from brief spikes while catching sustained degradation early. Based on the Google SRE multi-window approach.

Kubernetes Deployment

Helm-managed deployment with Datadog autodiscovery, pod anti-affinity, rolling updates, and graceful shutdown.

Key patterns: maxUnavailable: 0 ensures zero-downtime deploys. Pod anti-affinity spreads across both nodes and AZs. The preStop hook sleeps 5s before SIGTERM so the load balancer deregisters the pod first — no dropped connections. Datadog annotations enable automatic metric scraping and structured log collection.

Autoscaling (HPA)

Scales on both CPU utilization and custom p99 latency metric from Datadog.

Why custom metrics: CPU-only autoscaling misses I/O-bound latency issues. We scale up when p99 hits 400ms (80% of the 500ms SLO) to get ahead of breaches. Scale-down has a 5 minute cooldown to prevent flapping.

AU Region Configuration

Melbourne (ap-southeast-2) has different tuning than US due to cross-region latency.

Why different values: Cross-region calls to US-hosted services add ~150ms of latency. So the circuit breaker timeout is 45s (vs 30s US), failure threshold is 7 (vs 5), and the HPA latency target is 600ms (vs 400ms). Scale-down cooldown is 10 minutes to avoid scaling down before the AU morning traffic ramp.

Buildkite Deploy Pipeline

Build, test, canary, rollout — with manual gate before production and sequential US-then-AU deployment.

Pipeline design: Buildkite handles deploys (not GitHub Actions) because it supports concurrency_group to prevent simultaneous deploys, block steps for manual approval, and agent queues for environment isolation. The canary stage routes 10% of traffic to the new version and queries Datadog for 5 minutes before proceeding.

GitHub Actions (PR Validation)

Runs on every PR: tests, Helm lint, Terraform validate, security scans, Docker build.

Split CI strategy: GitHub Actions handles PR validation (fast feedback for developers). Buildkite handles deploys (production access, manual gates). This split means product engineers get CI feedback without needing Buildkite access.

ECS Fargate Service

Serverless container compute — runs alongside Kubernetes to demonstrate fluency with both.

Why both K8s and ECS: The role requires experience with both. The ECS module includes Fargate task definitions, ALB target groups with health checks, application autoscaling on CPU + request count, and a deployment circuit breaker that auto-rolls back if new tasks fail to stabilize.

Cloudflare Geo-Routing & WAF

Traffic routing, health checks, failover, WAF rules, and edge caching — all as Terraform.

Geo steering: Americas and Europe route to US. Asia-Pacific and Oceania route to AU. If one region fails health checks, all traffic fails over to the healthy region automatically. WAF rules rate-limit product API scraping and challenge empty user-agent requests. Edge caching gives product listings a 60-second TTL to absorb traffic spikes.
